9. Eu basicamente fiz um backend em padrão REST, onde cada rota tem uma responsabilidade específica (criar, listar, atualizar e deletar tarefas), sem misturar funções. Separei minimamente as coisas: as rotas cuidam da requisição/resposta, e a conexão com o banco fica isolada na função get_db(), o que já evita bagunça e facilita manutenção. O sistema é stateless, ou seja, não tem login nem sessão, cada requisição funciona de forma independente. Também usei MySQL como banco relacional pra garantir organização e consistência dos dados. Outra decisão foi usar PATCH dinâmico, permitindo atualizar só os campos enviados, deixando a API mais flexível. Também validei dados no backend (ex: impedir título vazio), não confiando só no frontend. No geral, fiz um backend simples, mas já estruturado pra escalar depois (tipo adicionar usuários futuramente sem precisar refazer tudo).

10. Eu garantiria a confiabilidade olhando pros aspectos de qualidade que já aparecem no meu código como a entrada no POST (não deixa criar tarefa com título vazio) e também trato casos inválidos no PATCH e DELETE (retornando 400 ou 404). Isso evita dado quebrado e deixa a API previsível. é válido ressaltar também que todas as rotas retornam status corretos (200, 201, 400, 404) e mensagens em JSON. Isso já ajuda muito a monitorar comportamento da API e identificar problemas rápido. Se eu evoluísse pra produção, adicionaria logs e métricas, mas mesmo assim já garanti uma base sólida de controle de erro e previsibilidade, que é essencial pra confiabilidade.

11. Se o sistema passasse a ter múltiplos usuários, eu teria que sair do modelo atual monousuário e adicionar autenticação, tipo login com JWT, pra identificar quem tá fazendo cada requisição. No banco, eu criaria uma tabela de usuários e relacionaria as tarefas com um idUsuario, porque hoje todas as tarefas são “globais”. Também teria que aplicar autorização, ou seja, garantir que cada usuário só consegue ver e mexer nas próprias tarefas, filtrando tudo pelo usuário logado. Além disso, eu criaria um middleware pra validar o token antes de acessar as rotas. Com isso, a arquitetura deixaria de ser simples e aberta e passaria a ter controle de acesso e separação de dados por usuário, além de criptogrfaia hash em todas as colunas, pois a segurança de dados dos clientes é questão prioritaria.
